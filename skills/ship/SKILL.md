# ship

作業中リポジトリの変更を本番反映まで届け、反映されたことを確認して報告する。

## 手順

1. **変更の把握**: `git status --short` と `git diff` で変更内容を確認する。依頼と無関係な変更が混ざっていたら、含めてよいかユーザーに確認する。
2. **安全確認**: ステージされる内容にAPIキー・トークン・`.env`系ファイル・認証情報が含まれていないか確認する。見つけたら**中断して報告**(コミットしない)。
3. **コミット**: Conventional Commits形式(feat: / fix: / docs: など)。ブランチ運用はグローバルCLAUDE.mdのルールに従う。
4. **push**: `git push`。
5. **デプロイ反映の確認**:
   - プロジェクトの `CLAUDE.md` / `AGENTS.md` に本番URL(`Live URL:` の行)があるか確認する。
   - **URLがあり、変更が画面に現れる場合**: 今回の変更で本番HTMLに新しく現れる文字列をマーカーとして選び、反映を待つ:
     ```bash
     until curl -s <LIVE_URL> | grep -q "<marker>"; do sleep 15; done; echo "deploy live"
     ```
     (バックグラウンド実行、目安10分でタイムアウト扱いにして状況報告)
   - **URLはあるが画面に現れない変更**(ドキュメント・設定のみ等): サイトがHTTP 200で稼働していることだけ確認する。
   - **URLがない場合**: pushまでで完了とし、その旨を報告する。
6. **見た目の確認**(UIに影響する変更のみ): Playwrightでスクリーンショットを撮って崩れがないか確認する(/screenshot-check スキルがあればそれを使う)。
7. **報告**: コミット内容の要約、本番URL、反映確認の結果を伝える。

## 注意

- push済みのコミットは取り消しにくい。手順1・2の確認を省略しない。
- 複数リポジトリに変更がある場合(例: スキル本体とサイト)、それぞれで手順を繰り返す。
